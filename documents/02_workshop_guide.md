## 0. はじめに ― 重要な前提（必ず最初に読む）

本手順書は当初「Gemini CLI の導入」を予定していましたが、**2026年6月18日をもって、無料・個人（Google AI Pro / Ultra 含む）向けの Gemini CLI（`gemini` コマンド）はサービスを終了しました。** 後継は **Antigravity CLI（`agy` コマンド）** です。

そのため本勉強会では **Antigravity CLI（`agy`）を導入・使用します。** 手順は第7章で詳述します。

| 項目 | 旧：Gemini CLI | 新：Antigravity CLI（本勉強会で使用） |
|------|---------------|----------------------------------|
| コマンド | `gemini` | `agy` |
| 個人/無料利用 | 2026/6/18 終了 | 利用可（無料枠あり） |
| インストール | `npm install -g @google/gemini-cli` | `curl -fsSL https://antigravity.google/cli/install.sh \| bash` |
| 無料枠の目安 | 1日 約1,000リクエスト | **1日 約20リクエスト（大幅縮小）** |

> ⚠️ **運営者への注意：無料枠は1日約20リクエストと非常に少ないです。** 参加者全員が同時にCLIを多用すると枠を使い切る可能性があります。第10章「運営上の注意」を必ず事前に確認してください。
>
> ※ Gemini CLI 自体は、組織で Gemini Code Assist Standard / Enterprise ライセンスや有料APIキーを持つ場合は継続利用可能です。自社がこれに該当する場合は、講師の判断で旧 Gemini CLI を使う選択肢もあります。

## 1. 実施目的

1. **バイブコーディングを「体験」として理解する**
   座学だけでなく、実際にAIに作らせ・公開し・直す一連を手を動かして体験する。
2. **AI時代の開発の基礎動作を身につける**
   自然言語での指示、Gitでの公開、CLIエージェントの利用、相互レビュー。
3. **AI出力を検証する姿勢を養う**
   「動かして確かめる」「鵜呑みにしない」を体で覚える。
4. **新人同士の交流・相互学習**
   作ったものを見せ合い、レビューし合うことで横のつながりをつくる。

## 4. 【ステップ②】ブラウザAIで静的コンテンツを作る

### 進め方
1. 参加者は ChatGPT / Claude / Gemini / Copilot のいずれかを開く。
2. 「作りたいもの」を自然言語で依頼する。**1ファイル（index.html）で完結する静的コンテンツ**を題材にする。
3. 出てきたコードをローカルに `index.html` として保存し、ブラウザで開いて動作確認する。

### お題の例
- HTMLだけで動くシューティングゲーム
- もぐらたたきゲーム
- ストップウォッチ／タイマー
- 自己紹介ページ（アニメーション付き）
- おみくじアプリ

### プロンプト例（そのまま使える）
```
HTMLとJavaScriptとCSSだけを使い、index.html 1ファイルで完結する
シューティングゲームを作ってください。
- 矢印キーで自機を左右に動かす
- スペースキーで弾を撃つ
- 敵が画面上から降りてくる
- 弾が敵に当たると敵が消えてスコアが増える
- 自機が敵に当たるとゲームオーバー
外部ライブラリは使わないでください。
コードには初心者向けにコメントを付けてください。
```

### ローカル保存と確認手順
1. 作業用フォルダを作る（例：`my-first-vibe`）。
2. AIの出力を `index.html` という名前で保存する。
3. そのファイルをダブルクリック、またはブラウザにドラッグして開く。
4. 動かない／思った通りでない場合は、**動作の様子を具体的に伝えて**AIに修正を依頼する。

> ポイント：この段階で「完璧」を目指さない。多少バグがあってOK。ステップ⑥でCLIを使って直すので、むしろ直しがいのある状態でよい。

## 5. 【ステップ③】Git で GitHub にアップロード

各参加者が**自分のGitHubアカウント**でリポジトリを作り、作品を上げます。

### 5-1. GitHub 上でリポジトリを作成
1. GitHub にログイン → 右上「＋」→ **New repository**
2. Repository name：`my-first-vibe`（任意）
3. **Public** を選択（GitHub Pages 公開のため）
4. 「Create repository」をクリック
5. 表示される `https://github.com/ユーザー名/my-first-vibe.git` をコピーしておく

### 5-2. ローカルから Git でアップロード
作業フォルダ（`index.html` がある場所）でターミナルを開き、順に実行します。

```bash
# 1. Gitリポジトリとして初期化
git init

# 2. ファイルをステージング
git add index.html

# 3. コミット（変更を記録）
git commit -m "初めてのバイブコーディング作品"

# 4. ブランチ名を main に
git branch -M main

# 5. GitHub上のリポジトリと接続（URLは自分のものに置き換え）
git remote add origin https://github.com/ユーザー名/my-first-vibe.git

# 6. アップロード
git push -u origin main
```

### 認証でつまずいたら
- HTTPSでpush時にパスワードを求められる場合、GitHubのパスワードでは通りません。
- **Personal Access Token（PAT）** を発行して、パスワード欄に貼り付けます。
  - GitHub → Settings → Developer settings → Personal access tokens → Tokens (classic) → Generate new token
  - スコープは `repo` にチェック
- 事前にメンターが発行手順をデモしておくと当日スムーズです。

### よくあるエラー
| メッセージ | 原因と対処 |
|-----------|-----------|
| `git: command not found` | Git未インストール。インストールが必要 |
| `Authentication failed` | PATを使う（上記参照） |
| `remote origin already exists` | `git remote remove origin` してから追加し直す |
| `failed to push (rejected)` | `git pull --rebase origin main` 後に再push |

## 6. 【ステップ④】GitHub Pages で公開・相互閲覧

### 6-1. 公開設定
1. リポジトリのページ → **Settings** → 左メニュー **Pages**
2. 「Build and deployment」→ Source を **Deploy from a branch** に
3. Branch を **main** / フォルダを **/(root)** に設定して **Save**
4. 数十秒〜数分後、`https://ユーザー名.github.io/my-first-vibe/` が発行される

### 6-2. 相互閲覧
- 各自の公開URLを、チャットや共有スプレッドシートに貼って一覧化する。
- 互いのURLにアクセスし、作品を見せ合う。
- 「良いと思った点」を1人1つメモしておく（ステップ⑧のレビューで使う）。

> 補足：反映に時間がかかることがあります。URLが404の場合は数分待ってリロード。

## 7. 【ステップ⑤】Antigravity CLI（agy）の導入 ★詳述

> ここが本勉強会の技術的な山場です。当初予定の Gemini CLI は提供終了したため、後継の **Antigravity CLI** を導入します。`gemini` ではなく `agy` コマンドを使う点に注意してください。

### 7-0. Antigravity CLI とは
- Google が提供する**ターミナル上で動くAIコーディングエージェント**。
- ファイルを読み書きし、コマンドを実行し、複数ステップの作業を自律的にこなせる。
- Gemini CLI の後継として位置づけられ、`agy` コマンドで起動する。
- macOS / Linux / Windows に対応。

### 7-1. インストール（OS共通の基本）

**macOS / Linux**
ターミナルで次の1行を実行します。インストーラが環境を判定し、`agy` バイナリを配置します。

```bash
curl -fsSL https://antigravity.google/cli/install.sh | bash
```

実行すると、おおむね次のように進みます。
```
✓ Platform detected: ...
✓ Latest available version: ...
✓ Download complete and checksum verified.
```

インストーラは `agy` をユーザーのローカル領域（Unix系では `~/.local/bin/`）に配置します。**PATHに含まれていない場合、追加すべき行が表示される**ので、その指示に従ってシェル設定ファイル（`~/.bashrc` や `~/.zshrc`）に追記してください。

**Windows**
- WSL（Windows Subsystem for Linux）上で、上記 macOS/Linux と同じコマンドを使うのが最も安定します。
- WSL未導入の場合は、ネイティブのインストール手順（公式ドキュメント参照）もありますが、当日は **WSL推奨**でメンターが補助する想定です。

### 7-2. ターミナルを再起動して動作確認
インストール後、**ターミナルを一度閉じて開き直し**、バージョンを確認します。

```bash
agy --version
```

バージョン番号が表示されれば成功です。`command not found` の場合は、インストール直後に表示されたPATH追記の指示が反映できているか確認してください（シェル設定ファイルへの追記＋ターミナル再起動）。

### 7-3. 初回起動と認証（Googleアカウント）
プロジェクトのフォルダ（`index.html` がある作業フォルダ）に移動してから起動します。

```bash
# 作業フォルダへ移動
cd my-first-vibe

# 起動
agy
```

初回起動時に **Google サインインのフロー**が始まります。
1. 認証方法を選ぶ（個人利用は **Google OAuth（Googleアカウント）** を選択）。
2. 表示されるリンクをブラウザで開く。
3. Googleアカウントでログインし、表示された**認証コード**をターミナルに貼り付ける。
4. 認証が完了すると、対話インターフェースが起動する。

> 注意：会社のGoogle Workspaceアカウントで認証制限がある場合は、個人のGoogleアカウントを使うか、事前にIT部門へ確認してください。

### 7-4. 基本的な使い方
- 起動後は、やりたいことを**自然言語で入力**するだけ。
- `/` を入力するとスラッシュコマンドの一覧が出る。
  - `/help` … ヘルプ
  - `/models` … 利用可能なモデル一覧
  - `/config` … 設定
- **権限設定**：はじめは安全側の **request-review（実行前に毎回確認）** のままにしておく。AIがファイルを書き換えたりコマンドを実行する前に確認が入るので、新人でも安心。
  - `always-proceed`（無確認実行）は、内容を理解できる環境でのみ使うこと。

### 7-5. 導入チェックリスト（参加者用）
- [ ] `agy --version` でバージョンが表示される
- [ ] `agy` を起動し、Google認証が完了している
- [ ] 作業フォルダ内で `agy` を起動できている
- [ ] 権限は `request-review` になっている

## 8. 【ステップ⑥⑦】CLIでデバッグ／ブラッシュアップ → 再アップロード

### 8-1. CLIでの依頼例（デバッグ）
作業フォルダで `agy` を起動した状態で、自然言語で依頼します。

```
このフォルダの index.html を読んで、ゲームのバグを直してください。
今、弾が敵に当たってもスコアが増えません。原因を調べて修正してください。
```

CLIが `index.html` を読み、原因を推定し、修正案を提示します（`request-review` なら適用前に確認が入る）。

### 8-2. CLIでの依頼例（ブラッシュアップ）
```
このゲームに次の機能を追加してください。
- スコアが10の倍数になるごとに敵の出現速度を上げる
- ゲームオーバー時に「リスタート」ボタンを表示する
変更後、何を変えたか日本語で要約してください。
```

### 8-3. 動作確認
- 修正後、必ずブラウザで `index.html` を開き直して動作を確認する。
- **「直ったと言われた」を信じず、自分の目で確かめる**（資料①の検証の習慣）。

### 8-4. 修正版を GitHub に再アップロード（ステップ⑦）
```bash
git add index.html
git commit -m "CLIでバグ修正と機能追加"
git push
```
GitHub Pages は自動で再ビルドされるので、数分後に公開URLへ反映されます。

## 9. 【ステップ⑧⑨】相互レビュー → 非公開化（後片付け）

### 9-1. 相互レビュー（ステップ⑧）
- 各自の**修正後**の公開URLにアクセスし合う。
- 観点を決めておくとレビューしやすい：
  - 動作するか／バグはないか
  - 操作はわかりやすいか
  - コードのコメントは読みやすいか（リポジトリも覗いてみる）
  - 「ブラッシュアップ前後でどう良くなったか」
- 1人1作品につき「良い点1つ・気づいた点1つ」を伝える形式にすると回しやすい。

### 9-2. 公開を非公開に戻す（ステップ⑨）
勉強会の作品を公開したままにしたくない場合の手順です。**いずれか**を実施します。

**方法A：GitHub Pages を無効化する**
1. リポジトリ → Settings → Pages
2. Source を **None** に変更して Save（公開停止）

**方法B：リポジトリ自体を Private にする**
1. リポジトリ → Settings → 一番下の **Danger Zone**
2. **Change repository visibility** → Private を選択
3. 確認のためリポジトリ名を入力して実行
> 注意：Privateにすると、無料プランでは GitHub Pages が公開されなくなります（＝実質非公開）。

**方法C：リポジトリを削除する**
1. Settings → Danger Zone → **Delete this repository**
2. 確認入力後に削除（**復元不可**なので、残したい場合は方法AかB）

> 後片付けの確認：非公開化後、別の人やシークレットウィンドウから公開URLにアクセスし、見えなくなっていることを確認すると確実です。

## 10. 運営上の注意（講師・メンター向け）

### ★ 無料枠が非常に少ない点への対策（最重要）
Antigravity CLI の無料枠は **1日あたり約20リクエスト**と少なく、参加者全員が同時に多用すると枠を使い切る恐れがあります。対策：

- **CLI操作（ステップ⑥）は依頼回数を絞る**よう事前案内する（例：1人あたり数回まで）。
- デバッグ・ブラッシュアップの**お題を事前に1〜2個に絞っておく**。
- 余裕があれば、**講師のデモ中心**にして全員一斉実行を避ける構成にする。
- 自社が **Gemini Code Assist の Standard / Enterprise ライセンス**や有料APIキーを持つなら、旧 Gemini CLI（`gemini`）の継続利用も検討（無料枠制限を回避できる）。

### その他の注意
- **インストールでのつまずきが最多**：PATHが通らず `command not found` になるケースが定番。ターミナル再起動とPATH追記の確認を最初に案内する。
- **認証で会社アカウント制限**に当たることがある → 個人Googleアカウントを案内。
- **Git認証（PAT）** は事前にデモしておく。当日最もつまずきやすい箇所のひとつ。
- ネットワーク：社内プロキシ環境ではインストーラやpushが弾かれることがある。事前に検証しておく。
- 仕様・コマンド・無料枠はツール側の都合で変わりやすい。**勉強会の直前に最新の公式情報を確認**すること。

### トラブル時の代替プラン
- CLIが使えない場合でも、**ステップ⑥のデバッグはブラウザAIでも代替可能**（コードを貼って直してもらう）。勉強会の本質（バイブコーディング体験）は維持できる。
