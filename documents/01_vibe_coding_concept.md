# バイブコーディング入門 ― 概要と概念

> 新人エンジニア向け勉強会 資料①
> 対象：エンジニア1〜2年目
> 目的：「バイブコーディング」を概念として理解し、体験への準備をする

---

## 1. バイブコーディングとは何か

**バイブコーディング（Vibe Coding）** とは、ソースコードを1行ずつ手で書くのではなく、**自然言語（日本語や英語の指示）でAIに「やりたいこと」を伝え、AIにコードを生成・修正させながら開発を進めるスタイル**のことです。

「こういう雰囲気（vibe）のものが欲しい」とAIに伝え、出てきた結果を見て、また言葉でフィードバックする。この対話の繰り返しでソフトウェアを形にしていきます。

### 従来の開発との違い

| 観点 | 従来の開発 | バイブコーディング |
|------|-----------|------------------|
| 主な入力 | プログラミング言語のコード | 自然言語の指示 |
| 開発者の役割 | コードを書く人 | やりたいことを定義し、AIの出力を評価・指示する人 |
| 試行錯誤の単位 | 関数・行 | 「お願い」と「フィードバック」 |
| 必要なスキルの比重 | 構文・APIの暗記 | 課題の言語化・判断・検証 |

### 言葉の由来

この言葉は2025年初頭にAI研究者のAndrej Karpathy氏が広めた表現が起点とされています。彼は「コードの存在をほとんど忘れ、AIに任せて雰囲気に身を委ねる」ような開発の感覚を表現しました。当初は遊び心のある言い回しでしたが、AIコーディング支援ツールの急速な普及とともに、開発スタイルを指す一般的な用語として定着していきました。

---

## 2. なぜ今バイブコーディングなのか

### 背景にある3つの変化

1. **大規模言語モデル（LLM）の性能向上**
   ChatGPT、Claude、Gemini などのモデルが、まとまった量の動くコードを一度に生成できるようになりました。

2. **対話型・エージェント型ツールの登場**
   単にコードを返すだけでなく、ファイルを読み書きし、コマンドを実行し、エラーを見て自分で直そうとする「エージェント型」のツールが普及しました（後述のCLIツールなど）。

3. **開発の重心が「実装」から「意図の伝達」へ**
   何を作るか・なぜ作るかを的確に言語化できる人の価値が相対的に高まっています。

### 向いている場面・向かない場面

**向いている場面**
- プロトタイプやアイデアの素早い検証
- 使い捨てのスクリプト、社内ツール、デモ
- 学習中の「動くものをまず触ってみたい」段階
- 定型的なコード（雛形、設定ファイル、テストデータ）の生成

**慎重に扱うべき場面**
- セキュリティや個人情報が絡む本番システム
- 高い正確性・再現性が求められる処理
- 大規模で長期保守が前提のコードベース
- 法的・倫理的な責任が重い領域

> ポイント：バイブコーディングは「考えなくていい」わけではありません。**何を作るかを考え、出てきたものが正しいかを判断する責任は人間側に残ります。**

## 3. バイブコーディングの基本サイクル

バイブコーディングは、おおむね次のループで進みます。

```
 ① 意図を言葉にする   →   ② AIに依頼する
        ↑                        │
        │                        ▼
 ④ 評価・追加指示    ←   ③ 出力を確認・実行する
```

1. **意図を言葉にする** … 「何を」「どう動いてほしいか」を具体的に書く
2. **AIに依頼する** … プロンプトとして渡す
3. **出力を確認・実行する** … 実際に動かして結果を見る
4. **評価・追加指示** … 良し悪しを判断し、修正点を言葉で返す

このループを速く・的確に回せることが、バイブコーディングの上達につながります。

---

## 4. 良いプロンプトを書くコツ

AIへの指示（プロンプト）の質が、出力の質を大きく左右します。新人のうちは特に次の点を意識してみてください。

### (1) 具体的に書く
- ✕「ゲームを作って」
- ◯「HTMLとJavaScriptだけで、矢印キーで自機を動かして弾を撃つシューティングゲームを1ファイルで作って。敵は上から降りてきて、当たるとゲームオーバーにして」

### (2) 制約・前提を伝える
使う技術、ファイル構成、対応ブラウザ、外部ライブラリの可否などを明示します。
- 例：「外部ライブラリは使わず、index.html 1枚で完結させて」

### (3) 役割・ゴールを与える
- 例：「あなたは初心者向けの教材を作る講師です。コードにコメントを多めに付けて」

### (4) 段階的に進める
一度に全部を求めず、「まず動く最小版を作って」→「次にスコア表示を足して」と小さく刻みます。

### (5) フィードバックは具体的に
- ✕「なんか違う」
- ◯「弾の速度が速すぎるので半分に。あと自機が画面外に出てしまうので、画面端で止まるようにして」

---

## 5. 出力を鵜呑みにしないために（検証の習慣）

AIは**もっともらしいが間違ったコード**を出すことがあります。これを「ハルシネーション」と呼ぶこともあります。新人のうちから次の習慣をつけましょう。

- **必ず動かして確かめる** … 「動くはず」で終わらせない
- **エラーメッセージを読む** … そのままAIに貼って原因を尋ねるのも有効
- **存在しないAPI・ライブラリに注意** … AIがそれらしい嘘の関数名を作ることがある
- **セキュリティ・個人情報** … パスワードやAPIキーを安易に貼らない、生成コードに危険な処理がないか確認する
- **コピペ前に「自分の言葉で説明できるか」** … 説明できないコードは理解できていないサイン

> 合言葉：**「AIは優秀な新人アシスタント」**。仕事は任せられるが、最終チェックと責任は自分が持つ、という距離感がちょうど良いです。

## 6. 体験で使う主なツール（この勉強会で触れるもの）

### ブラウザで使う対話型AI
| ツール | 提供元 | 特徴 |
|--------|--------|------|
| ChatGPT | OpenAI | 最も広く使われている対話型AI |
| Claude | Anthropic | 長文・コード生成に強い |
| Gemini | Google | Google サービスとの連携 |
| Copilot | Microsoft | Microsoft / GitHub と連携 |

→ ステップ②では、これらに「自然言語でお願いするだけ」で静的コンテンツ（HTMLゲーム等）を作ってもらいます。

### ターミナルで使うエージェント型CLIツール
ブラウザのチャットと違い、**自分のPCのファイルを直接読み書きし、コマンドも実行できる**のが特徴です。デバッグやブラッシュアップ（ステップ⑥）で威力を発揮します。

> ⚠️ **重要な注意（2026年6月時点）**
> 当初予定していた **Gemini CLI（`gemini` コマンド）は、2026年6月18日をもって無料・個人向けの提供を終了**し、後継の **Antigravity CLI（`agy` コマンド）** に置き換わりました。本勉強会では後継の **Antigravity CLI** を使います。詳しい導入手順は資料②を参照してください。

## 7. この勉強会のゴール

この勉強会を終えたとき、参加者が次の状態になっていることを目指します。

- バイブコーディングが「何をするものか」を自分の言葉で説明できる
- ブラウザのAIに指示して、動く静的コンテンツを1つ作れる
- 作ったものを Git で GitHub に上げ、GitHub Pages で公開できる
- エージェント型CLIツール（Antigravity CLI）を導入し、コードの修正に使える
- AIの出力を「鵜呑みにせず検証する」感覚を持てる
